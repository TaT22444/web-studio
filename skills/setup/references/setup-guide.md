# セットアップガイド（Claude内部参照用）

> このファイルは `/web-studio:setup` の実行ガイドです。
> 成果物は以下の4ファイルに書き出します。
> - `{project}/docs/interview.md`（Q&A生ログ）
> - `{project}/docs/context.md`（事業背景・判断基準）
> - `{project}/docs/requirements.md`（仕様・構成）
> - `{project}/docs/DESIGN.md`（デザイン方向性ドラフト）

---

## 起動時の判定

まず以下のファイルの存在を確認し、モードを決定する。

| 状態 | モード |
|---|---|
| `docs/interview.md` がない | フルセットアップ（Stage 1〜3） |
| `docs/interview.md` はある（途中再開） | interview.md を読み込み、未回答ステージから再開 |
| `docs/context.md` と `docs/requirements.md` がある | 更新モード → ユーザーに選択肢を提示 |

**途中再開の方法:**
`interview.md` を読み込み、どのステージまで回答が記録されているかを確認する。
記録済みのステージは再度聞かない。未記録のステージから再開する。

**更新モードの選択肢:**
```
どちらを更新しますか？
1. 全体をやり直す
2. 事業・目的だけ更新する（context.md）
3. 仕様・要件だけ更新する（requirements.md）
4. デザイン方向性だけ更新する（DESIGN.md）
```

---

## Q&A フロー（フルセットアップ）

### ⚠ 重要：各ステージ完了後に即座にファイルへ書き込む

ステージごとの回答を受けたら、**次のステージへ進む前に必ず `interview.md` へ追記する**。
これにより、会話が長くなっても「どこまで終わったか」をファイルで確認できる。

書き込みのタイミング:
- Stage 1 回答受け取り → `interview.md` に Stage 1 を書き込む → Stage 2 へ
- Stage 2 回答受け取り → `interview.md` に Stage 2 を追記 → Stage 3 へ（必要なら）
- Stage 3 回答受け取り → `interview.md` に Stage 3 を追記 → Stage 4 へ
- Stage 4 完了 → `context.md` / `requirements.md` / `DESIGN.md` を書き出す

---

### Stage 1「事業・目的・背景」

事業の前提とサイトの役割を1度にまとめて聞く。

> プロジェクトの前提をまとめて教えてください。
> ここで伺った内容がデザインや構成の判断軸になります。
> 決まっていない項目は「未定」「おまかせ」で大丈夫です。
>
> **事業について**
> - どんな事業・サービスをされていますか？（誰に何を届けているか）
> - 個人のプロジェクトですか、クライアント向けですか？
> - 何を作りますか？（例: Webサイト / LP / コーポレートサイト）
> - サイト名・プロジェクト名はありますか？（仮でOK）
>
> **目的・背景**
> - このサイトの目的は何ですか？（集客・信頼構築・情報発信・採用・販売など）
> - サイトを通じて実現したいことは？（具体的な成果があれば）
> - なぜ今このタイミングで作りますか？
> - 制約や絶対に避けたいことはありますか？（なければ「なし」でOK）

**回答受け取り後、すぐに `interview.md` の Stage 1 セクションへ書き込む。**

---

### Stage 2「ターゲット・デザイン・構成」

ターゲット、デザインの方向性、構成を1度にまとめて聞く。

> 次に、サイトの具体的な方向性を教えてください。
> 決まっていない項目は「未定」「おまかせ」で大丈夫です。こちらから提案します。
>
> **ターゲットと行動**
> - 主なターゲットユーザーは？
> - ユーザーに取ってほしい行動（CTA）は？（問い合わせ・購入・来店・資料請求など）
>
> **デザインの方向性**
> - サイト全体の雰囲気・世界観のイメージは？（落ち着いた / 活気ある / 洗練 / 温かみ / クール 等）
> - 使いたい色やブランドカラーはありますか？（なければ「おまかせ」でOK）
> - 参考になるサイトやデザインはありますか？（URLでも「こんな感じ」でもOK）
> - 避けたい印象は？（安っぽい / 堅すぎる / うるさい / 地味 等）
> - 写真や画像の雰囲気は？（人物 / プロダクト / 風景 / 抽象的 等）
> - レイアウトの好みは？（余白多め / 情報密度高め / 写真大きめ 等）
>
> **構成と技術**
> - 入れたいページや機能はありますか？（決まっているものだけでOK）
> - 技術的な要望はありますか？（なければこちらで提案します）

**回答受け取り後、すぐに `interview.md` の Stage 2 セクションへ追記する。**

> 💡 ページ構成が固まっていない場合は「未定」で構いません。Stage 3 でディレクターが提案します。

---

### Stage 3「追加ヒアリング（必要な場合のみ）」

Stage 1〜2 の回答を読み返し、サイト制作を進める上で「ここが不明だと設計できない」という箇所があれば、**最大3問だけ**追加で聞く。
全員に必ず聞くのではなく、「これが不明だと構成が決められない」という項目だけを選ぶ。
不明点がなければこのステージはスキップしてよい。

#### 追加質問を行う条件（どれかに当てはまるとき）
- CTAが曖昧で「何をもって成功か」が分からない
- トンマナが抽象的すぎて、デザイン方向を決める根拠がない
- 既存素材（文章/画像/ロゴ）の有無が不明で、制作体制に影響しそう

#### 質問の方針
- **最大3問**に絞る
- 可能なら選択肢形式で聞く
- スキップ判断をした場合は「追加質問なし、Stage 3 へ進みます」と明示する

**回答受け取り後（または スキップの場合）、すぐに `interview.md` の Stage 3 セクションへ追記する。**

---

### Stage 4「ディレクター分析・戦略整理」

Stage 1〜3 の回答（`interview.md` を参照）をすべて踏まえ、ディレクターとして分析を行い、戦略インサイトを整理する。
**このステージでセクション構成やレイアウトは決めない。** それはワイヤーフレームフェーズでAIデザイナーが判断する。
ここで整理するのは「**何を伝えるべきか・誰に・なぜ**」という判断の根拠。

#### 4-1. 情報の整理と分析

ヒアリング内容を整理し、以下をチャットに表示する:

- **ユーザー像の具体化:** ターゲットがサイトに来るときの状況・気持ち・求めている情報
- **CTA到達までの心理フロー:** 訪問 → 理解 → 信頼 → 行動 の各段階でユーザーが必要とするもの
- **最優先で伝えるべきこと:** ファーストビューで伝えるべき核心は何か
- **コンテンツ優先順位:** 事業の強み・ユーザーの関心に基づく情報の重み付け
- **差別化ポイント:** 競合・業界と比較してこのサイトが強調すべきこと

#### 4-2. ユーザー確認

分析を示し、認識にずれがないか確認する:
- 「このサイトで一番伝えたいこと」の理解は合っているか
- ターゲット像の解釈に違和感はないか
- 必須コンテンツに漏れはないか

ずれがあれば修正し、**承認を得てから出力フォーマットへ進む。**

> セクション構成・レイアウトについては「ワイヤーフレームで提案しますね」と一言添えて次へ進む。
