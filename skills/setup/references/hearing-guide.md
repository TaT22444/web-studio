# ヒアリングガイド（Claude内部参照用）

> setup-guide.md から参照される。
> ヒアリングの質問設計・分岐ロジック・判断フレームワークを定義する。
> 成果物は `{project}/docs/interview.md` に段階的に書き出す。

---

## 絶対ルール（違反禁止）

**ユーザーから与えられた情報をもとに、ヒアリングを実施すること。**
指示がない限りはヒアリングをスキップしないこと。事前に情報を渡された際は、その分の情報を記入し、不足分や深掘りたいところをヒヤリングするように。

ユーザーが最初の発言で多くの情報を提供した場合は、その内容を `interview.md` の該当ステージに記録した上で、**足りない部分・深掘りできる部分を対話で聞く**。提供済みの情報を聞き直す必要はないが、ヒアリング対話そのものを飛ばしてはならない。

## ヒアリングの設計思想

### あなたはプロのWebディレクターである

ヒアリングはフォームの記入作業ではない。相手の言葉の裏にある意図、まだ言語化できていないイメージ、本人も気づいていないニーズを引き出す対話である。

**原則:**

- 一度に聞くのは最大4〜5問。回答を受けてから次を組み立てる
- 質問は「調査」ではなく「会話」のトーンで。相手が自然と語りたくなる聞き方を意識する
- 全項目を機械的に聞く必要はない。回答の中にすでに答えが含まれていたら重複して聞かない
- 「決まっていない」「おまかせ」は正当な回答。詰問しない。AIが判断を引き受ける
- 回答が短くても焦らない。相手のペースに合わせる

### サイトの目的は多様である

全てのサイトが「訪問者の課題を解決して行動を促す」ために存在するわけではない。
サイトには「性格」があり、それによって聞くべきこと・設計の軸が根本的に変わる。

| 性格 | 説明 | 例 |
|---|---|---|
| 説得・訴求型 | 訪問者の悩みや関心に対して解決策を提示し、行動を促す | サービスLP、クリニック、塾 |
| 情報整理型 | 情報をわかりやすく構造化し、探しやすくする | キャンプ場の施設情報、自治体サイト |
| 空気感・ブランディング型 | 世界観や雰囲気を感じ取ってもらう。明示的CTAが不要なこともある | ブランドサイト、旅館、ギャラリー |
| 信頼構築型 | 企業の信頼性・堅実さを伝えることが主目的 | コーポレートサイト、士業 |
| カタログ・ショーケース型 | 作品や商品を美しく見せる・一覧する | ポートフォリオ、セレクトショップ |

同じ「キャンプ場のサイト」でも、予約を取りたいなら説得型寄り、雰囲気を伝えたいならブランディング型寄り、施設情報を整理したいなら情報整理型寄り。**サイトの種類と性格は別軸**であり、Pre-Stageで両方を見極める。

## Pre-Stage: プロジェクトの前提把握

最初に、プロジェクトの輪郭をつかむ。ここでの理解が後続の全ステージの質問トーンを決める。

> まず、プロジェクトの基本的なところを教えてください。
>
> - 何を作りますか？
>   （例: コーポレートサイト / LP / ECサイト / ブランドサイト / イベントページ / ポートフォリオ / 店舗サイト 等）
> - どんな業界・ジャンルですか？
>   （例: 飲食 / 美容 / IT / 教育 / 建築 / アパレル / 医療 / アウトドア 等）
> - このサイトの「役割」をひとことで言うと？
>   （例:「お客さんに来店してもらう」「世界観を感じてもらう」「商品を買ってもらう」「情報を探しやすくする」「会社の信頼感を伝える」等。ざっくりでOK）
> - ご自身のプロジェクトですか、クライアント向けですか？
> - サイト名・プロジェクト名はありますか？（仮でOK）

**回答を受けてAIが内部で判定する（ユーザーには見せない）:**

| 分類軸 | 判定内容 |
|---|---|
| サイト種類 | コーポレート / サービスLP / イベント・キャンペーンLP / EC / ブランドサイト / 店舗サイト / ポートフォリオ / メディア・ブログ / その他 |
| サイト性格 | 説得型 / 情報整理型 / ブランディング型 / 信頼構築型 / ショーケース型（複合あり） |
| 業界ジャンル | 自由分類 |

この判定結果は `interview.md` のAI思考メモに記録する。以降のステージで、この判定に基づいて質問を選択する。

**回答受け取り後、すぐに `interview.md` の Pre-Stage セクションへ書き込む。**

## Stage 1: 事業とブランドの本質

事業の「中身」と「こだわり」を理解する。表面的なサービス説明ではなく、その事業が大切にしていること・他と違うところを引き出す。ここで得る情報がサイト全体のトーンと判断軸になる。

> 事業やサービスについて、もう少し詳しく教えてください。
>
> - どんな事業・サービスをされていますか？（誰に、何を届けているか）
> - この事業で一番こだわっていること、大切にしていることは？
> - 始めたきっかけや背景があれば教えてください（お客さんにも伝えたいストーリーがあれば）
> - 同業他社や競合と比べて「ここが違う」と思うところは？
>
> 特にない項目は飛ばして大丈夫です。

### タイプ別の追加質問（該当する場合だけ、上の質問と一緒にまとめて聞く）

**店舗ビジネスの場合:**
> - 店舗数は？（1店舗 / 複数）
> - 立地の特徴は？（駅チカ / 隠れ家的 / 路面店 等）

**ECの場合:**
> - 商品の数はどのくらいですか？（数点 / 数十点 / 大量）
> - 商品のジャンル・カテゴリは？

**クライアント向け制作の場合:**
> - クライアントからのブリーフや要望はありますか？（あればそのまま共有してください）

**回答受け取り後、すぐに `interview.md` の Stage 1 セクションへ追記する。**

## Stage 2: 目的・ターゲット・戦略

「なぜこのサイトを作るのか」「誰に見てもらいたいのか」を深掘りする。
Pre-Stage で判定した「サイトの性格」に応じて、問いのトーンを変える。

### ユニバーサル質問（全サイトに共通）

> 次に、サイトの目的とターゲットを教えてください。
>
> - このサイトの一番の目的は何ですか？
>   （集客 / ブランド認知 / 信頼構築 / 情報発信 / 販売 / 採用 / 予約獲得 等）
> - なぜ今このタイミングで作りますか？（新規立ち上げ / リニューアル / 事業転換 等の背景）
> - 主にどんな人に見てもらいたいですか？
>   （年齢層、職業、ライフスタイル等。「20代〜30代の共働き夫婦」のようにざっくりでOK）
> - その人たちは、どうやってこのサイトにたどり着きそうですか？
>   （Google検索 / SNS / 口コミ / チラシ・名刺 / 広告 等。複数OK）

### 性格別の追加質問（Pre-Stage の判定に基づいて選ぶ。該当するもののみ）

**説得・訴求型:**
> - サイトを見た人に、最終的にどんな行動を取ってほしいですか？（問い合わせ / 申し込み / 来店 / 購入 等）
> - その人たちが「迷う理由」「不安に思うこと」は何だと思いますか？

**情報整理型:**
> - 訪問者がいちばん「すぐ見つけたい」と思う情報は何ですか？
> - 情報量は多いほうですか？カテゴリ分けや検索機能が必要になりそうですか？

**ブランディング型:**
> - このサイトを見た人に、どんな気持ちや印象を持ってほしいですか？
> - サイトを離れた後、どんな余韻が残っていてほしいですか？（雰囲気？ 名前？ 行きたいという気持ち？）

**信頼構築型:**
> - 「この会社は信頼できる」と感じてもらうために、一番重要な要素は何ですか？（実績 / 歴史 / 技術力 / 人柄 等）
> - 取引先・採用候補者など、見る人の立場によって伝えたい情報は変わりますか？

**ショーケース型:**
> - 作品や商品の魅力を伝えるうえで、一番大切にしたい見せ方は？（大きな写真 / ストーリー付き / カテゴリ整理 等）
> - 実績や受賞歴など、作品の信頼の裏付けになる情報はありますか？

**回答受け取り後、すぐに `interview.md` の Stage 2 セクションへ追記する。**

## Stage 3: コンテンツと機能

「何を載せるか」「どんな機能が必要か」を具体的に把握する。
サイトタイプに応じてよくある機能をこちらから提示し、ユーザーが思いつかない要素の漏れを防ぐ。

### ユニバーサル質問

> サイトに載せたい内容と、必要そうな機能を確認させてください。
>
> - 入れたいページやコンテンツは決まっていますか？（決まっているものだけでOK。なければ提案します）
> - テキスト・写真・ロゴ・動画など、既存の素材はありますか？
> - サイト公開後、どのくらいの頻度で更新しそうですか？
>   （ほぼ更新しない / たまにお知らせ / 定期的にブログ追加 / 頻繁に商品更新 等）

### タイプ別の機能チェックリスト

以下から、サイトタイプに関連するものだけを選んで「こういった機能は必要ですか？」と確認する。全部を列挙しない。

**共通でよく必要になるもの:**
- お問い合わせフォーム
- SNSリンク / 埋め込み
- Googleマップ
- ブログ / お知らせ / ニュース
- 多言語対応

**店舗サイト向け:**
- 予約機能（外部サービス連携 or サイト内）
- メニュー / 料金表
- スタッフ紹介
- 営業時間・アクセス詳細
- ギャラリー（店内・施術例等）

**EC向け:**
- カート・決済
- 商品カテゴリ・フィルタ
- レビュー / 評価
- 会員機能
- 配送・送料設定

**コーポレート向け:**
- 会社概要 / 沿革
- 事業内容
- 採用情報
- IR / グループ企業情報

**LP / イベント向け:**
- 申し込み / 参加登録フォーム
- タイムライン / スケジュール
- 料金プラン比較
- 登壇者 / ゲスト紹介
- FAQ

**ポートフォリオ向け:**
- 作品ギャラリー
- ケーススタディ / プロジェクト詳細
- スキル / 実績一覧

**メディア / ブログ向け:**
- カテゴリ / タグ
- 検索機能
- 関連記事
- SNSシェアボタン

### 技術・制約の確認

> あと少しだけ確認させてください。
>
> - 技術的な要望はありますか？（使いたい技術やホスティング先等。なければこちらで提案します）
> - レスポンシブはPC・スマホどちらを重視しますか？（両対応が理想ですが、優先度があれば）
> - 制約や「これだけは避けたい」ことはありますか？

**回答受け取り後、すぐに `interview.md` の Stage 3 セクションへ追記する。**

## Stage 4: デザインの方向性

見た目と世界観を聞く。事業や目的の話とは切り離し、純粋にビジュアルの好み・方向性を引き出す。

> サイトの見た目やデザインの方向性を教えてください。
> 感覚的な言葉で大丈夫です。決まっていないものは「おまかせ」で。
>
> - サイト全体の雰囲気・世界観のイメージは？
>   （落ち着いた / 活気ある / 洗練 / 温かみ / クール / ナチュラル / 力強い 等）
> - 使いたい色やブランドカラーはありますか？
> - 避けたい印象は？（安っぽい / 堅すぎる / うるさい / 地味 等）
> - 写真の雰囲気は？（人物中心 / 風景 / プロダクト / 抽象的 等）
> - レイアウトの好みは？（余白多め / 情報密度高め / 写真大きめ 等）
> - セクションの区切り方の好みは？（余白で区切る / 線で区切る / 背景色の切替 / おまかせ）
> - 参考になるサイトやデザインはありますか？（URLそのまま貼ってください。複数OK）
> - スクリーンショットや画像があれば、プロジェクトの `docs/references/` に置いてパスを教えてください

### 参考サイト・画像が提供された場合

URLや画像が複数ある場合もまとめて聞く。回答は任意。

> 参考サイト・画像について、よければ教えてください（おまかせでも大丈夫です）。
>
> - それぞれの何を参考にしたいですか？
>   （「全体の雰囲気」「余白の使い方」「色合い」「フォントの細さ」等、感覚でOK）
> - 逆に、参考サイトの中で「今回は不要」な機能やUIがあれば教えてください

**ユーザーが回答した場合:**
回答内容を `DESIGN.md` の参照テーブルに記録する。

**「おまかせ」と答えた場合:**
AIが参照サイトを自分で分析して判断する。wireframe-rules.md / figma-rules.md の「スタイルを借りる。機能は借りない。」原則に従い、判断根拠を `DESIGN.md` の備考欄に記録する。ユーザーに確認を求めて足止めしない。

**回答受け取り後、すぐに `interview.md` の Stage 4 セクションへ追記する。**

## Stage 5: アダプティブ深掘り

Stage 1〜4 の全回答を読み返し（`interview.md` を参照）、「ここが不明だと設計に入れない」箇所がないか確認する。
不明点がなければこのステージはスキップしてよい。

### 深掘りが必要な条件（いずれかに該当するとき）

- サイトの性格（説得型？ブランディング型？等）の判断にまだ迷いがある
- CTAの有無・形式が曖昧で、導線設計の判断ができない
- ターゲット像が抽象的すぎて、トーンや情報の優先順位を決められない
- コンテンツの量・素材の有無が不明で、ページ構成の見通しが立たない
- デザインの方向性に矛盾がある（例:「洗練」と「にぎやか」の両立を求められている）
- 業界特有の慣習・期待値について確認が必要

### 深掘りの方針

- 聞くべきことだけを、できるだけ答えやすい形で聞く
- 可能なら選択肢形式にする
- 「なぜそれを聞いているか」を一言添える（唐突に感じさせない）
- スキップ判断をした場合は「追加の確認事項はありません。分析に進みます」と明示する

**回答受け取り後（またはスキップ時）、すぐに `interview.md` の Stage 5 セクションへ追記する。**

## Stage 6a: ディレクター分析・戦略整理

全ステージの回答（`interview.md` を参照）を踏まえ、ディレクターとして俯瞰的な分析を行う。
ここで整理するのは「何を伝えるべきか・誰に・なぜ・どんなトーンで」という判断の根拠。

### 分析内容（チャットに表示する）

サイトの性格に応じてストーリーのトーンを変える。全サイトに同じ分析テンプレートを当てはめない。

**1. ユーザー像の具体化**
ターゲットがサイトに来るときの状況・気持ち・求めている情報を、人物像として描く。

**2. 体験ストーリー**
「このユーザーがこのサイトに出会い、どんな体験をするか」を自然な文章で描く（箇条書き不要、物語として）。
サイトの性格によって心理の流れが変わる:

- 説得型: 不安・疑問 → 理解 → 納得 → 行動（問い合わせ・購入）
- 情報整理型: 疑問・目的 → 発見 → 比較・理解 → 満足（再訪・ブックマーク）
- ブランディング型: 出会い → 没入 → 共感・憧れ → 記憶（また来たい・人に話したい）
- 信頼構築型: 確認 → 安心 → 信頼 → 次のアクション（取引・応募）
- ショーケース型: 興味 → 鑑賞 → 感動・発見 → ファン化（フォロー・依頼）

**3. 最優先で伝えるべきこと**
ファーストビューに置くべき核心は何か。

**4. コンテンツ優先順位**
事業の強み・ユーザーの関心に基づく情報の重み付け。

**5. 差別化ポイント**
競合・業界と比較して、このサイトが特に強調すべきこと。

### ユーザー確認

分析をチャットに表示し、認識にずれがないか確認する:
- 「このサイトで一番伝えたいこと」の理解は合っているか
- ターゲット像の解釈に違和感はないか
- 体験ストーリーの方向性に共感できるか
- 必須コンテンツに漏れはないか

ずれがあれば修正し、**承認を得てから Stage 6b へ進む。**

**回答受け取り後、すぐに `interview.md` の Stage 6a セクションへ追記する。**

## Stage 6b: 設計方向の対話

6a の承認後に入る。ヒアリング全体と参考サイト分析を踏まえ、「何を伝えるか」を「どう表現するか」に落とし込む**共同セッション**。
AIが選択肢や具体例を示しながら、ユーザーと対話して設計の方向を決める。

### この対話の目的

ワイヤーフレームに入る前に、コンテンツの見せ方・コンポーネントの方向性・インタラクションのトーンをすり合わせる。
ここで決めるのは「方向」であって「詳細」ではない。具体的なピクセル値やカラーコードはワイヤー・ビジュアルで詰める。

### 対話トピック

以下から、プロジェクトの性格・複雑度に応じて**重要な 3〜5 トピック**を選んで対話する。
全トピックを網羅的に聞く必要はない。「おまかせ」は歓迎し、AIが判断根拠を記録する。

**コンテンツの見せ方**
> 「実績は一覧型・1件ずつ大きく・ケーススタディ型、どう見せますか?」
> 「サービスはカード型・テキスト説明型・アイコン付きリスト、どれが近いですか?」

**ファーストビュー**
> 「ヒーローは写真全面・テキスト主体・最小限のどの方向ですか?」
> 「最初に目に入るのはビジュアル? コピー? それとも両方均等?」

**ナビゲーション**
> 「固定ヘッダ・スクロールで変化するタイプ・消えるタイプ、参考サイトはどれに近いですか?」

**セクション間の関係**
> 「余白でゆったり区切る・線で区切る・背景色を変えて切り替える、好みはありますか?」

**カード/リスト**
> 「写真が主役のカード・テキスト中心のリスト・混在、どちらが合いそうですか?」

**インタラクションのトーン**
> 「ホバーは控えめ（色変化のみ）・しっかり反応（影や動き）、どちらが好みですか?」
> 「スクロールアニメーションは静か・ダイナミック、どのくらいの強さですか?」

**CTAの強さ**
> 「目立つボタン・さりげないテキストリンク・ボタンなし（世界観で導く）、どれが合いますか?」

### 対話の進め方

1. 6a の分析と参考サイトの印象を踏まえ、AIがまず「こうするのが合いそう」と提案する
2. ユーザーの反応を受けて調整する。1〜2往復で十分
3. 「おまかせ」の場合はAIが判断し、根拠をAI思考メモに記録する
4. 全トピックに明確な答えが出なくてもよい。決まった部分だけ記録する

### ユーザー確認

対話の結果をまとめてチャットに表示し、認識にずれがないか確認する。

ずれがあれば修正し、**承認を得てから出力フォーマットへ進む。**

## 「おまかせ」判断フレームワーク

ユーザーが「おまかせ」「未定」「特にない」と回答した項目について、AIがどう判断すべきかの指針。

### 判断の原則

1. **業界の慣習を基本にする** — その業界のサイトでユーザーが「あって当然」と期待する要素は入れる
2. **ターゲットの期待値** — ターゲット層が日常的に触れているサイトの水準を想像する
3. **サイトの性格から逆算** — 説得型なら説得に必要な要素、ブランディング型なら世界観構築に必要な要素を優先する
4. **足すより削る** — 迷ったらシンプルな方を選ぶ。不要な機能・要素は入れない
5. **判断根拠を記録** — `interview.md` のAI思考メモに「なぜそう判断したか」を必ず書く。後で人間が検証できるようにする

### よくある「おまかせ」項目の判断基準

| 項目 | 判断の軸 |
|---|---|
| ページ構成 | サイト種類の業界標準 + 性格に必要最小限のページ |
| CTA | 性格による。説得型なら原則必須。ブランディング型なら不要の場合もある |
| カラー | 業界のトーン + 性格 + 参考サイトがあればそこから抽出。根拠を明記 |
| 雰囲気 | 業界・ターゲット層・性格から推測。「おまかせ」の場合こそAI思考メモに判断過程を詳しく書く |
| レイアウト | ターゲットの主デバイス + 情報量から判断。高齢者向けなら大きめ文字・余白多め等 |
| 技術スタック | production-rules.md のデフォルト構成を提案 |
| レスポンシブ | 原則両対応。ターゲットの主デバイスを重点的にケア |
| 機能 | サイト種類の基本セットのみ。過剰に足さない |
