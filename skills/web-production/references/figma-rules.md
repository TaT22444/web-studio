# 🎨 ビジュアルデザイン出力ルール（Claude内部参照用）

> Phase 3 の成果物は以下のいずれかで書き出します。
> - **1ページのみ:** `{project}/design-draft.html`
> - **複数ページあり:** `{project}/design-draft/` フォルダ以下にページごとのHTMLファイル
> ワイヤーフレーム（Phase 2）が承認された後にのみ実行する。

---

## このフェーズでのあなたの役割

あなたは今、Web制作会社のシニアデザイナーです。
クライアントのことを深く理解し、そのブランドに命を吹き込む仕事をします。

Phase 3 の成果物は「静的なモックアップ」ではなく **「リビングプロトタイプ」** です。
色やレイアウトだけでなく、動き・リズム・インタラクションまで含めて体験を設計してください。
クライアントや実装者がブラウザで開いたとき、「これが完成形だ」と感じるレベルを目指す。

「ルールに沿った正解」を出すのではなく、「このプロジェクトにしかない表現」を考えてください。

## デザイン着手前に全ファイルを読み込む

以下を必ず読み込んでから、**一切コードを書く前に**設計フェーズに入ること。

1. `{project}/docs/context.md` — 事業背景・AIの判断基準
2. `{project}/docs/requirements.md` — ターゲット・ユーザーの体験ストーリー
3. `{project}/docs/DESIGN.md` — 世界観・カラー方向性・**参考サイトURL・参考画像パス**・Do's/Don'ts
4. `{project}/wireframe.html` または `{project}/wireframe/` 以下の**全ファイル** — 承認済みの構造とインタラクション意図
5. `{project}/docs/references/` — 参考画像・スクリーンショット・ブランドアセット（存在する場合）

**DESIGN.md に参照URLが記載されている場合は、必ずそのURLを直接フェッチして視覚分析する。**
言語化された説明文を読むだけでは不十分。実際のサイトを見て：
- 配色（背景・テキスト・アクセントの関係性）
- タイポグラフィ（フォントの質感・ウェイト・字間）
- コンポーネントの形・余白・深度
- セクション間の流れとリズム

を直接把握してからデザインを始める。「このサイトから何をどう借りるか・何を独自にするか」を言語化してからコードに入る。

**`docs/references/` に画像ファイルが存在する場合は、URLフェッチと同等の優先度で読み込む。**
スクリーンショットはURLフェッチより高精度な視覚情報を含む場合がある（レンダリング後の実際の見た目）。
画像から以下を読み取り、DESIGN TOKENS コメントに反映する：
- 背景と文字のコントラスト比・明度関係
- フォントのウェイト感・字間・行間の雰囲気
- カードや画像のトリミング・角丸・シャドウの有無
- ナビゲーションのスタイル（透過/背景色/高さ）
- ボタン・CTAの形状とトーン

### ⚠ 参照分析の鉄則：「スタイルを借りる。機能は借りない。」

参照サイトに存在する機能・UIコンポーネントは **そのサイトのビジネス要件から生まれたもの** であり、今回のサイトとは無関係。
どれだけ印象的な機能でも、今回のサイトに同じ理由がなければ実装しない。

✅ **借りてよいもの（スタイル・トーン・視覚的判断）**
- 配色・コントラスト・明度・彩度のバランス
- フォントのウェイト感・字間・行間・サイズ比率
- 余白の量・呼吸感・情報密度
- レイアウトの構図（非対称・グリッド・フルブリード等）
- アニメーション・インタラクションのトーン（速さ・イージング・質感）
- 写真の扱い（全幅・クロップ・モノクロ・彩度処理等）
- hover / scroll / transition の演出スタイル

❌ **借りてはいけないもの（機能・コンテキスト依存の要素）**
- そのサイト固有のUI機能（表示モード切替・3D/2D切替・ダークモード・フィルター・検索・会員機能・カート等）
- ナビゲーションの項目数・構造（そのサイトのコンテンツ量・サイト規模に依存している）
- ページネーション・無限スクロール等の閲覧補助機能（今回のコンテンツ量に不要なら実装しない）
- CTAの形式・文言（今回のビジネスゴールと一致する理由がある場合のみ参考にする）

**判断の軸：「この機能はなぜそのサイトに存在するか？今回のサイトに同じ理由があるか？」**
DESIGN.md の「このサイトでは不要な要素」欄に記載がある場合は、それを最優先で守る。

**ワイヤーフレームから読み取るのは「意図」であり、「構造」ではない。**

ワイヤーフレームを見て理解すべきことは：
- このページで伝えるべき情報は何か
- ユーザーにどんな体験・心理の流れを設計しているか
- どんなアクションが必要でどこに配置する意図があったか

セクションのレイアウト・コンポーネントの形・余白の使い方・動きの演出はすべてデザイナーであるあなたが判断する。ワイヤーの構造に縛られる必要はない。

**複数ページのワイヤーがある場合は、同じページ数分のデザインファイルを作成する。**

## デザインを始める前に自問する

コードを書く前に、以下の問いに答えること。答えが出てから手を動かす。

**ブランドの本質**
- このサイトを見た人に、最初の3秒で何を感じてほしいか？
- このブランドを一言で言うと何か？それをビジュアルと動きでどう表現するか？
- DESIGN.md の世界観・参考サイトから、どんな「空気感」が見えるか？

**ターゲットへの共鳴**
- このサイトを訪れるユーザーはどんな人で、何を期待しているか？
- このサイトを見た後、ユーザーにどうなってほしいか？
- 明示的なCTAボタンが必要か？不要な場合、目的をどうビジュアルと体験で達成するか？
- requirements.md の体験ストーリーを読んで、どんな感情の流れを設計するか？

**動きと体験の設計**
- スクロールはどんな体験であるべきか？（静かに展開する / ダイナミックに動く / 没入する）
- アニメーションはブランドの個性を強化するか？（高速 = 活発 / 遅め = 高級感 / 控えめ = 洗練）
- どこに「驚き」や「発見」を仕掛けるか？
- インタラクション（hover, scroll, load）はどう設計するか？

**このプロジェクト固有の表現**
- 参考サイトの「好きな点」は何か？その本質を取り込みつつ、どう独自にするか？
- 避けたいこと（Do's/Don'ts）を踏まえて、何を「やらない」と決めるか？

---

## デザインの意思決定

上記の問いへの答えを踏まえ、以下を自分で決定する。

- **カラーパレット**: DESIGN.md の方向性を基に、Primary・Secondary・Background・Accent・Text を決定。根拠を DESIGN TOKENS コメントに記録する
- **タイポグラフィ**: サイズ・ウェイト・字間・行間の体系。見出しの「顔」がブランドの第一印象になる
- **余白とリズム**: どこを「息を吸う場所」にして、どこを「密度で引き込む場所」にするか
- **背景の変化**: セクションをどんな色・質感でつなぐか。スクロールの旅として設計する
- **深度と質感**: shadow・border・gradient・透過でどこに「奥行き」を作るか
- **動きのトーン**: ブランドに合う速度・イージング・距離感を統一する
- **アクセント**: ブランドの個性が最も光る場所を1〜2箇所に絞って強く使う

## アニメーション・インタラクション設計

### 表現手法の選択

プロジェクトの世界観に合った手法を選ぶ。以下は手法の例であり、全部使う必要はない。
**このブランドに何が最も合うか**で判断する。

| 手法 | 向いているブランド |
|---|---|
| パララックス（背景と前景の速度差） | 奥行き・高級感・没入感が必要なとき |
| スクロールフェードイン | 上品に情報を展開したいとき。ほぼ全サイトで有効 |
| ページロードアニメーション | ブランドの第一印象を強く演出したいとき |
| ホバーマイクロインタラクション | 親しみやすさ・遊び心・精密感を出したいとき |
| スクロール連動テキストアニメ | ストーリーテリング・没入感が強いコンテンツ |
| ナビゲーションのスクロール変化 | ほぼ全サイトで入れるべき基本の動き |
| カウントアップアニメ（数字） | 実績・数字を印象的に見せたいとき |
| 画像の視差 / クリッピング | アーティスティック・エディトリアルな表現に |

### 実装方針

- **CSS アニメーション・トランジション**: `<style>` タグに記述。hover効果・エントランス・ローディングに
- **Intersection Observer**: スクロールで要素が入ったときにクラスを付与してアニメーション発動
- **スクロールイベント**: パララックス・ナビ変化・プログレスバーなどに
- **CDN ライブラリ**: デザインプロトタイプなのでGSAP・AOS・Lenis などの使用を許可。ただし使いすぎない

```html
<!-- CDN 使用例（必要な場合のみ） -->
<script src="https://cdnjs.cloudflare.com/ajax/libs/gsap/3.12.2/gsap.min.js"></script>
<script src="https://cdnjs.cloudflare.com/ajax/libs/gsap/3.12.2/ScrollTrigger.min.js"></script>
```

## ダミー画像の使い方（picsum.photos）

グレーのプレースホルダーボックスは使わない。**実際のダミー写真を `img` タグで配置すること。**

```
https://picsum.photos/seed/{キーワード}/{幅}/{高さ}
```

`seed` にキーワードを入れると同じキーワードで毎回同じ画像が返る。カード複数枚は `item-01` / `item-02` のようにシードを変えて別々の画像にする。

---

## 技術的制約

| # | ルール |
|---|---|
| 1 | **1ページ = 1 HTMLファイル**。1ページなら `design-draft.html`、複数ページなら `design-draft/` フォルダ |
| 2 | **`<style>` タグ許可** — CSSはHTMLの `<head>` 内に `<style>` タグでまとめて記述 |
| 3 | **`<script>` タグ許可** — JSは `</body>` 直前にまとめて記述 |
| 4 | **CDN 許可**（アニメーションライブラリ等。プロトタイプ用途のため） |
| 5 | **画像は `picsum.photos` のダミー写真を使用** |
| 6 | **ワイヤーフレームと同じページ数・同じファイル名で出力する** |
| 7 | **必ず上書き（write）。追記（append）禁止。** `<!DOCTYPE html>` は1回のみ |

## フォント

**Google Fonts を積極的に使用すること。** フォント選択はブランドの印象を大きく左右する最重要デザイン判断の一つ。

```html

<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=..." rel="stylesheet">
```

DESIGN.md の参考サイト・世界観・雰囲気から判断して選ぶ。参考サイトが使っているフォントに近いものを選ぶのも有効。

## 出力チェックリスト

**デザインの意図:**
- [ ] DESIGN TOKENS コメントに「デザインのコンセプト」と「アニメーションのトーン」が書かれているか
- [ ] カラー・タイポ・余白の選択に DESIGN.md・context.md からの根拠があるか
- [ ] ブランドの個性が最も出ているアクセントポイントが1〜2箇所あるか
- [ ] ユーザーの体験ストーリーに沿った感情の流れがビジュアルと動きに反映されているか

**アニメーション・インタラクション:**
- [ ] スクロールでページを下げたとき、「生きている」と感じる動きがあるか
- [ ] ナビゲーションのスクロール変化が実装されているか
- [ ] アニメーションの速度・イージングがブランドの世界観と一致しているか

- [ ] hover など、操作したときの反応が設計されているか

**技術:**
