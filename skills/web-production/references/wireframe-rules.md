## ワイヤーフレームフェーズの役割

**レイアウト・セクション構成・情報設計の決定権はここにある。**

ユーザーから提供されるのは「何を伝えたいか・誰に・どんな雰囲気で」という情報だけ。
「どんなレイアウトにするか・セクションをいくつにするか・どう見せるか」はAIデザイナーが判断する。
requirements.md にセクション構成が書かれていたとしても、それは「叩き台」に過ぎない。最終的なレイアウト判断はここで行う。

以下のファイルをすべて読み込み、**このプロジェクト固有の最適解**を導く：
- `context.md` → 事業・目的・AIの判断基準
- `requirements.md` → ターゲット・必須コンテンツ・戦略インサイト
- `DESIGN.md` → 世界観・レイアウト好み・余白感・**参考サイトURL・参考画像パス**

**参照URLが記載されている場合は、必ずそのURLを直接フェッチして視覚的に分析する。**
言語化された説明文だけを読むのではなく、実際のサイトを見てレイアウト・情報設計・コンポーネント構造・余白感を把握する。
「このサイトのどの部分をワイヤーに活かすか」を自分の言葉で整理してから作業を開始する。

**`docs/references/` に画像ファイルが存在する場合は、必ず読み込んで視覚分析する。**
URLフェッチと同様に扱う。スクリーンショット・ロゴ・ブランドアセットを含む。
画像から読み取った情報（構図・余白・コンポーネント形状・タイポグラフィの重さ等）を、ワイヤーの設計根拠コメントに反映する。

### ⚠ 参照分析の鉄則：「スタイルを借りる。機能は借りない。」

参照サイトに存在する機能・UIコンポーネントは **そのサイトのビジネス要件から生まれたもの** であり、今回のサイトとは無関係。
視覚的な美しさに引きずられて、今回のサイトに不要な機能を実装してはならない。

✅ **借りてよいもの（スタイル・トーン・美的判断）**
- 配色・コントラスト・明度のバランス
- フォントのウェイト感・字間・サイズ比率
- 余白の量・セクションの呼吸・情報密度
- レイアウトの構図（非対称・グリッド・フルブリード等）
- アニメーション・インタラクションのトーン（速さ・動きの質感）
- 写真の扱い（全幅・クロップ・モノクロ処理等）
- セクション間のリズム変化

❌ **借りてはいけないもの（機能・コンテキスト依存の要素）**
- そのサイト固有のUI機能（表示モード切替・3D/2D切替・フィルター・検索・会員機能・ダークモード等）
- ナビゲーションの項目数・構造（そのサイトのコンテンツ量に依存している）
- CTAの形式・文言（今回のビジネスゴールと一致する理由がある場合のみ参考にする）
- コンテンツ量・情報密度の模倣（業種・規模が違えば適切な密度も異なる）

**判断の軸：「この機能はなぜそのサイトに存在するか？今回のサイトに同じ理由があるか？」**
理由がなければ、どれだけ印象的な機能でも実装しない。
DESIGN.md の「借りない要素」欄に記載がある場合は、それを最優先で守る。

## 設計思想：「ページはストーリーである」

ワイヤーフレームはレイアウトテンプレートの流し込みでも、要件の箇条書きをHTMLにしたものでもない。
`context.md`・`requirements.md`・`DESIGN.md` を深く読み解き、**ユーザーを説得するストーリー**として構成する。

### レイアウト設計の前に答えるべき問い

ワイヤーフレーム作成前に、以下を自問してから構成に入ること。

1. **ユーザーは何を求めてサイトに来るか？** → ファーストビューに何を置くべきか
2. **ユーザーにどんな行動を取ってほしいか？** → 明示的なCTAボタンが必要か？来店・記憶・シェアが目的なら「ボタンなし」でよい。目的に合った導線設計をする
3. **信頼を得るために何が必要か？** → 実績・お客様の声・資格等の配置判断
4. **情報の優先順位は？** → セクション順序の根拠
5. **業界やターゲットの期待値は？** → 慣習に沿うか、あえて外すかの判断

## 情報設計の原則（最重要）

### レイアウトパターンのデフォルト化を避ける

AIは学習データから「よくある構成」を無意識に選ぶ傾向がある。
以下のパターンは一見自然に見えるが、何も考えずに選ぶと陳腐になる：

| よくやりがちなデフォルト | 問題点 |
|---|---|
| Hero = 左テキスト + 右画像の2カラム | 全サイトが同じに見える |
| サービス紹介 = 必ず3カードグリッド | 情報量に関わらず3列になる |
| About = 左画像 + 右テキストの2カラム | 業種関係なく同じパターン |
| CTA = センタリングバナー | インパクトが薄い |

**context.md・requirements.md・DESIGN.md を読み、「このサイトにとって最適な構成」を導く。**
デフォルトに近い構成を選ぶ場合も、「なぜそれが最適か」を WIREFRAME STRATEGY コメントに必ず書く。

---

### 「要件の項目数 = セクション数」にしない

requirements.md に 10 個の情報項目があっても、10 セクション作ってはいけない。
関連する情報はグルーピングし、1つのセクションに統合する。

悪い例：
```
[Hero] → [会社概要] → [理念] → [事業内容] → [強み] → [実績] → [チーム] → [お客様の声] → [サービス] → [料金] → [FAQ] → [アクセス] → [お問い合わせ] → [Footer]
```

良い例：
```
[Hero — 価値提案 + CTA]
[サービス概要 — 事業内容 + 強みを統合]
[実績数字 — 薄いバンドで3つの数字]
[お客様の声 — カード2〜3枚]
[About — 会社概要 + 理念 + チームを統合]
[CTA — お問い合わせ誘導]
[Footer — アクセス + 基本情報]
```

### 情報に強弱をつける

全セクションが「全幅 + 見出し + 本文」の同じパターンの繰り返しにしない。
情報の重要度に応じてセクションの面積・密度・レイアウトパターンを変える。

| 情報の重要度 | 表現方法 |
|---|---|
| 最重要（Hero・メインCTA） | 大きく、余白多め、画面占有率高い |
| 重要（サービス・強み） | カード型・グリッドで密度ある表現 |
| 補強（実績数字・受賞歴） | 薄いバンド・アクセント的に |
| 信頼（お客様の声・パートナー） | 引用 + 写真、コンパクトに |
| 行動喚起（CTA） | シンプルに1行 + ボタン |
| 補足（FAQ・アクセス） | 畳める構造、または Footer に統合 |

### ページ全体のリズムを設計する

セクションの「密度」「レイアウトパターン」「高さ」を意図的に変え、緩急をつける。
同じパターンが3回以上連続しないようにする。

リズムの例：
```
[大きく・余白多め] Hero — 呼吸
[密度あり・カード] サービス — 情報
[薄い・数字だけ]  実績 — アクセント
[引用・写真あり]  お客様の声 — 信頼
[シンプル・1行]   CTA — 行動
```

### ストーリーで考える

ページを「情報のリスト」ではなく「説得のシナリオ」として設計する。
ユーザーがスクロールしながら辿る心理の流れ：

```
興味（Hero: この会社/サービスは何？）
  ↓
理解（サービス: 何を提供してくれるのか）
  ↓
納得（実績・強み: なぜここを選ぶべきか）
  ↓
信頼（お客様の声: 本当に大丈夫か）
  ↓
行動（CTA: 問い合わせよう）
```

この心理フローに沿わない情報配置は、ユーザーの離脱につながる。

## サイト性格別の視覚構成

`requirements.md` の「サイト性格」を確認し、その性格にふさわしい **視覚的な構成** を目指す。
以下はテンプレートではなく「語彙」。同じ性格でもプロジェクトごとに構成は異なるが、目指すべき方向を理解するために使う。

### ブランディング型

**目指すもの**: 世界観への没入。サイト自体がブランド体験になる。

- ヒーローはビューポート全体を使い、写真やビジュアルが主役。テキストは最小限（店名 + 一行コピー程度）
- 情報は「提示する」のではなく「発見される」ように配置する。スクロールして初めて見えるコンテンツがあってよい
- セクション見出しは名詞ラベル（「コンセプト」「メニュー」）ではなく、感情や世界観を表す言葉にする
- 写真の面積 > テキストの面積。写真のサイズ差・配置のリズムがデザインそのもの
- 余白を「贅沢に」使う。詰め込まない勇気が高級感を生む
- フルブリードセクションと狭幅テキストセクションの交互で呼吸感を作る

避けること: 全セクションにラベル見出しをつけて一覧化すること。均等なカードグリッドの多用。情報の網羅性を優先してしまうこと。

### 説得型（サービスLP等）

**目指すもの**: 明確なストーリーフローで、ユーザーを行動に導く。

- ヒーローは価値提案 + CTA が軸。「何を解決するか」が最初の3秒で伝わる
- セクション順序が「心理の階段」になっている: 興味→理解→証拠→信頼→行動
- 数字（実績）は独立した薄いバンドで目立たせると効果的
- お客様の声・事例は「信頼のピーク」として適切な位置に配置する
- CTA は繰り返し登場してよい（ヒーロー + ページ途中 + 末尾）

避けること: 情報を均等に並べてしまうこと。CTAの位置が末尾にしかないこと。

### 情報整理型（コーポレート等）

**目指すもの**: 信頼感と網羅性。必要な情報に迷わずたどり着ける。

- 整然とした構造は正しい選択。ただし密度の緩急で退屈を避ける
- ナビゲーションが重要。ページ数が多い場合、回遊性を設計する
- 数字・ファクトを効果的に使い、テキストだらけにならないようにする
- 見出しは明快なラベル型でよい（このタイプではそれが適切）

避けること: 全セクションが同じ「見出し + 本文 + カード3枚」の繰り返し。密度が均一で単調になること。

### ショーケース型（ポートフォリオ・店舗等）

**目指すもの**: ビジュアルが最大面積。「見たい・行きたい・体験したい」を写真で喚起する。

- 写真ギャラリーがページの最大面積を占める。テキストは写真の補助
- 写真のサイズ差（大+小の組み合わせ）がリズムを作る。均等グリッドは避ける
- 実用情報（営業時間・アクセス等）は「必要な時に見つかる」程度に控えめに
- ヒーローは全面写真 + 最小テキストか、スライドショー的な構成

避けること: 写真を小さなカードに閉じ込めること。実用情報をビジュアルと同格で大きく扱うこと。

## 設計根拠とインタラクション意図の記録

セクション構成・順序の根拠と、デザインフェーズへのインタラクション指示をHTMLコメントに記録する。
このコメントが Phase 3 デザイナーへの「ブリーフ」になる。

### ページ全体の戦略コメント（冒頭に必須）

```html
<!--
  WIREFRAME STRATEGY: [サイト名]

  ターゲット: [誰が、どんな状況で見るか]
  主目的: [ユーザーに何をしてもらいたいか]

  ページストーリー:
  - [心理フロー: 興味→理解→信頼→行動 等]

  セクション設計:
  1. Hero — [なぜこのヒーロー構成か]
  2. [セクション名] — [目的 + なぜこの位置か + 統合した情報]
  3. [セクション名] — [目的 + なぜこの位置か]
  ...

  リズム設計:
  - [大→密→薄→引用→シンプル 等の緩急パターン]

  情報の統合:
  - [何と何を1セクションにまとめたか、その理由]

  インタラクション方針:
  - [このページ全体の動きのトーン: 静か / ダイナミック / 没入感 等]
  - [スクロール体験の設計意図]
-->
```

### セクションごとのインタラクション意図（各セクション冒頭に記録）

Phase 3 デザイナーが「どんな動きを設計すべきか」を迷わないよう、各セクションに意図を書く。
アニメーションを設計するのではなく、「何を感じさせたいか・何が必要か」を伝える。

```html
<!-- ─────────────────────────────────────
  セクション: Hero
  目的: 世界観への没入。最初の3秒で「ここは違う」と感じさせる
  インタラクション意図:
    - ページロード時にテキストが段階的に現れる（ブランドの静けさを演出）
    - 背景画像はスクロールに連動して奥へ引いていく（パララックス）
    - スクロールインジケーターで「続きがある」を示す
  優先度: 最高 — ここが離脱か継続かの分岐点
───────────────────────────────────────── -->
<section id="hero" ...>

<!-- ─────────────────────────────────────
  セクション: Philosophy
  目的: 「この店は本物だ」という確信を静かに与える
  インタラクション意図:
    - スクロールで入ってきたときにテキストがゆっくり浮き上がる
    - 装飾ライン（縦線など）がアニメーションで伸びると効果的
  優先度: 中 — ブランドへの信頼形成。派手な動きは逆効果
───────────────────────────────────────── -->
<section id="philosophy" ...>
```

### インタラクション意図の記述ガイド

| 記述すべきこと | 記述しなくていいこと |
|---|---|
| このセクションで何を感じさせたいか | 具体的なCSSプロパティや数値 |
| スクロール時・ロード時・ホバー時の意図 | アニメーションの実装方法 |
| 動きの優先度（必須 / あると良い / 不要） | フレームワークやライブラリの指定 |
| 動きの「速度感・トーン」のイメージ | コードの詳細 |

## 絶対ルール（違反禁止）

| # | ルール |
|---|---|
| 1 | **グレースケールのみ使用**。色の判断をこのフェーズに持ち込まない |
| 2 | **CSSは全てインライン記述**（`style=""`属性のみ）。`<style>`タグ・外部CSSファイル禁止 |
| 3 | **1ページ = 1 HTMLファイル**。1ページのみなら `wireframe.html`、複数ページなら `wireframe/` フォルダにページごとのファイルを作成する（後述の命名規則を参照） |
| 4 | **外部リソース読み込み禁止**。CDN・Google Fonts・外部画像URL禁止 |
| 5 | **JavaScriptは禁止**。`<script>`タグ禁止 |
| 6 | **`<link>`タグ禁止**（faviconも含む） |
| 7 | **ファイルは必ず上書き（write）する。追記（append）禁止。** 修正・再生成時も、完全なHTML文書1つだけで上書きすること。`<!DOCTYPE html>` が2回以上出現するファイルは壊れたファイルである |

## 複数ページの場合のファイル構成

`requirements.md` に複数のページが定義されている場合、各ページを独立した HTML ファイルとして出力する。

### ファイル命名規則

| ページ | ファイル名 |
|---|---|
| トップ / ホーム | `wireframe/index.html` |
| About / 会社概要 | `wireframe/about.html` |
| サービス | `wireframe/services.html` |
| 料金 | `wireframe/pricing.html` |
| ブログ / ニュース | `wireframe/blog.html` |
| お問い合わせ | `wireframe/contact.html` |
| その他 | `wireframe/{ページ名（英小文字）}.html` |

### ページ間ナビゲーション

各ページの上部に、シンプルなナビゲーションバーを設置してページ間を移動できるようにする。
リンクは同じ `wireframe/` フォルダ内の相対パスで記述する（例: `<a href="./about.html">`）。

```html
<!-- ナビゲーション例 -->
<nav style="background: #333; padding: 12px 40px; display: flex; gap: 24px;">
  <a href="./index.html" style="color: #fff; text-decoration: none; font-size: 14px;">TOP</a>
  <a href="./about.html" style="color: #fff; text-decoration: none; font-size: 14px;">About</a>
  <a href="./services.html" style="color: #fff; text-decoration: none; font-size: 14px;">Services</a>
  <a href="./contact.html" style="color: #fff; text-decoration: none; font-size: 14px;">Contact</a>
</nav>
```

### WIREFRAME STRATEGY コメントの書き方（複数ページ）

トップページ（`index.html`）のコメントにサイト全体の戦略を記載し、各サブページにはそのページ固有の目的を記載する。

## カラー（グレースケール内で自由に）

グレースケールのみ使用する。ただし決まったパレットに縛られる必要はない。
白（`#FFFFFF`）から黒（`#000000`）の間で、**明度差を使って情報の階層・重要度・リズムを表現する**。

判断基準:
- 背景の切り替え方はサイトの性格で決める。白→薄グレーの交互だけが正解ではない。ダークな背景セクション、グラデーション的な濃淡の変化なども有効
- テキストのコントラストは読みやすさを確保した上で、強弱をつける
- 要素の強調方法（濃い背景 + 白文字、ボーダー、シャドウ等）はレイアウトに応じて選ぶ

> 色を排除することで、レビュー時に「構造と情報の優先度」だけに集中できる。
> ただし `DESIGN.md` に記載されたレイアウトの好み（余白感・密度・画像比率）は反映する。

## HTML構造ルール

### ボイラープレート

`<!DOCTYPE html>`, `<head>`, `<meta>` は以下の通り。`<body>` のスタイリングはサイトの性格に合わせて自分で決める。

```html
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>[サイト名] - ワイヤーフレーム</title>
</head>
<body style="margin: 0; padding: 0; ...自分で決める...">
  <!-- WIREFRAME STRATEGY コメント -->
  <!-- セクションを配置 -->
</body>
</html>
```

### セクション構造

- 各セクションは `<section>` タグで囲み、`id` 属性を付与する
- セクションごとに `style=""` でレイアウトを完結させる
- **各セクションの冒頭にHTMLコメントで「このセクションの意図」と「統合した情報」を記録する**
- ネストは最大4階層まで

```html
<!-- 意図: サービス内容と強みを統合。ユーザーが「何を得られるか」を一目で理解できるようにする -->
<section id="services" style="...">
  ...
</section>
```

## コンテンツルール

### テキスト

- 見出し・CTA・キャッチコピーは**実際に近い文言**を入れる（Lorem ipsum 禁止）
- 本文テキストは内容の方向性が分かる程度に記載する（完璧な原稿は不要）
- **文字量を実際に近づける**ことで、レイアウトの現実性を検証する
- **見出しは「名詞ラベル」にしない。** 「メニュー」「会社概要」「アクセス」のようなラベル型見出しは情報の一覧に見える。サイト性格に応じて、キャッチコピー型・問いかけ型・世界観を表す言葉を選ぶ。ただし情報整理型サイトでは明快なラベル見出しが適切な場合もある

### 画像プレースホルダー

画像が入る場所にはグレーのプレースホルダーを配置し、**どんな画像を入れるべきかをテキストで明記する**。
「画像」とだけ書くのではなく、「笑顔のスタッフが作業している写真」のように意図を伝える。

プレースホルダーのサイズ・形状・配置はレイアウトに合わせて自分で決める。
ヒーロー全面の写真、カード内のサムネイル、背景画像など、用途に応じた適切なサイズにする。

### 画像プレースホルダーの配置義務（空欄禁止）

- ヒーローセクションには、必ず画像プレースホルダーを配置する
- 写真・図・実物イメージが必要になりそうなセクションでは、空欄にしない
- カード型セクションでは、各カードに画像枠を用意する

### アイコン

シンプルなSVGで直接記述する。外部アイコンフォント禁止。

## タイポグラフィ

フォントサイズの固定レンジは設けない。**サイトの性格とブランドのトーンに基づいて自分で決める。**

判断基準:
- **ヒーロー見出し**: ブランドの第一印象を決める最重要要素。力強いブランドなら大きく太く、繊細なブランドなら軽く控えめに
- **セクション見出し**: ヒーローとの差で階層を作る。ヒーローが大きいほど、セクション見出しとの差も大きくなる
- **本文**: 可読性が最優先。14px を下限の目安とする
- **全体**: 重要なのはサイズの「差」による階層が明確であること。使うサイズの種類は少ないほどまとまる

## レイアウト

固定の max-width や padding は設けない。**サイトの性格とセクションの役割に基づいて自分で決める。**

### コンテンツ幅の判断基準

- **フルブリード**: ヒーロー写真、雰囲気を伝えるセクション、ビジュアル主体のギャラリーなど → 画面幅いっぱい
- **ワイド**: 情報量が多いセクション、カードグリッドなど → 1000〜1400px 程度
- **ナロー**: テキスト中心の読み物、引用、コンセプト文など → 600〜800px 程度
- 1ページ内で幅を変えることでリズムが生まれる。全セクション同じ幅にしない

### セクション間の余白の判断基準

- セクション間の余白はそのセクションの役割と前後の関係で決める
- ヒーローの直後、CTAの直前など「呼吸」が必要な場所は広く
- 関連する情報が続く場所は狭めて連続感を出す
- 全セクション同じ padding にしない。余白の差がリズムになる

### グリッド・スペーシング

- グリッドは Flexbox で表現する
- スペーシングは 8 の倍数系を推奨するが、必須ではない。見た目の「心地よさ」を優先する

### レスポンシブ

ワイヤーフレームはPCビューのみで出力する。
ただし、レスポンシブで崩れにくい構造を意識する:

- `max-width` + `margin: auto` でセンタリング
- `flex-wrap: wrap` でカード等が折り返せる構造にする

## 出力チェックリスト

出力前に以下を必ず確認する。

**視覚構成の質（最重要）:**
- [ ] このサイト性格（ブランディング/説得/情報整理/ショーケース）にふさわしい視覚構成になっているか
- [ ] 全セクションが同じ幅・同じ padding・同じパターンの繰り返しになっていないか
- [ ] フルブリードと狭幅のセクションを使い分けているか（画面幅のリズム）
- [ ] セクション間の余白に差があるか（間の取り方のリズム）
- [ ] ブラウザで開いた時に「プロが作ったWebサイト」として違和感がないか

**構造・ストーリー:**
- [ ] WIREFRAME STRATEGY コメントが冒頭にあり、インタラクション方針が書かれているか
- [ ] 各セクションの冒頭コメントに「目的」と「インタラクション意図」が書かれているか
- [ ] 「要件の項目数 = セクション数」になっていないか（情報が統合されているか）
- [ ] 同じレイアウトパターンが3回以上連続していないか（リズムがあるか）
- [ ] セクションごとに面積・密度が変わっているか（強弱があるか）
- [ ] ページ全体がストーリー（興味→理解→信頼→行動）として成立しているか
- [ ] requirements.md の「ユーザーに期待する次の行動」に沿った導線設計になっているか

**インタラクション意図:**
- [ ] Hero など第一印象の強いセクションに「ページロード時の動き」の意図が書かれているか
- [ ] スクロール体験全体の「トーン・速度感」が WIREFRAME STRATEGY に書かれているか
- [ ] 各セクションにインタラクションの優先度（必須 / あると良い / 不要）が書かれているか
- [ ] ホバー状態が必要なコンポーネント（カード・ボタン等）にその旨が記録されているか

**コンテンツ:**
- [ ] グレースケール以外の色を使っていないか
- [ ] テキストが Lorem ipsum ではなく実際の内容に近いか
- [ ] 画像プレースホルダーに具体的な画像意図が書かれているか
- [ ] 画像領域があるセクションでは placeholder が入っているか（空欄禁止）

**技術:**
- [ ] `<style>`タグが存在しないか
- [ ] `<link>`タグが存在しないか
- [ ] `<script>`タグが存在しないか
- [ ] 外部URLが含まれていないか
- [ ] 全てのCSSが `style=""` 属性で記述されているか
- [ ] 1ページなら `wireframe.html`、複数ページなら `wireframe/` フォルダに分割されているか
- [ ] 複数ページの場合、各ページにナビゲーションが設置されているか
- [ ] 各ファイル内で `<!DOCTYPE html>` が1回だけ存在するか（複数回 = 追記ミス）
