# 📐 ワイヤーフレーム出力ルール（Claude内部参照用）

> このファイルはClaudeが Phase 2 を進めるための内部ガイドです。
> Phase 2の成果物は `{project-name}/wireframe.html` に書き出します。
> 要件定義（Phase 1）が承認された後にのみ実行する。

---

## ワイヤーフレームフェーズの役割

**レイアウト・セクション構成・情報設計の決定権はここにある。**

ユーザーから提供されるのは「何を伝えたいか・誰に・どんな雰囲気で」という情報だけ。
「どんなレイアウトにするか・セクションをいくつにするか・どう見せるか」はAIデザイナーが判断する。
requirements.md にセクション構成が書かれていたとしても、それは「叩き台」に過ぎない。最終的なレイアウト判断はここで行う。

以下のファイルをすべて読み込み、**このプロジェクト固有の最適解**を導く：
- `context.md` → 事業・目的・AIの判断基準
- `requirements.md` → ターゲット・CTA・必須コンテンツ・戦略インサイト
- `DESIGN.md` → 世界観・レイアウト好み・余白感・画像比率の方向性

---

## 設計思想：「ページはストーリーである」

ワイヤーフレームはレイアウトテンプレートの流し込みでも、要件の箇条書きをHTMLにしたものでもない。
`context.md`・`requirements.md`・`DESIGN.md` を深く読み解き、**ユーザーを説得するストーリー**として構成する。

### レイアウト設計の前に答えるべき問い

ワイヤーフレーム作成前に、以下を自問してから構成に入ること。

1. **ユーザーは何を求めてサイトに来るか？** → ファーストビューに何を置くべきか
2. **ユーザーにどんな行動を取ってほしいか？** → CTAの配置と導線設計
3. **信頼を得るために何が必要か？** → 実績・お客様の声・資格等の配置判断
4. **情報の優先順位は？** → セクション順序の根拠
5. **業界やターゲットの期待値は？** → 慣習に沿うか、あえて外すかの判断

## 情報設計の原則（最重要）

### レイアウトパターンのデフォルト化を避ける

AIは学習データから「よくある構成」を無意識に選ぶ傾向がある。
以下のパターンは一見自然に見えるが、何も考えずに選ぶと陳腐になる：

| よくやりがちなデフォルト | 問題点 |
|---|---|
| Hero = 左テキスト + 右画像の2カラム | 全サイトが同じに見える |
| サービス紹介 = 必ず3カードグリッド | 情報量に関わらず3列になる |
| About = 左画像 + 右テキストの2カラム | 業種関係なく同じパターン |
| CTA = センタリングバナー | インパクトが薄い |

**context.md・requirements.md・DESIGN.md を読み、「このサイトにとって最適な構成」を導く。**
デフォルトに近い構成を選ぶ場合も、「なぜそれが最適か」を WIREFRAME STRATEGY コメントに必ず書く。

---

### 「要件の項目数 = セクション数」にしない

requirements.md に 10 個の情報項目があっても、10 セクション作ってはいけない。
関連する情報はグルーピングし、1つのセクションに統合する。

悪い例：
```
[Hero] → [会社概要] → [理念] → [事業内容] → [強み] → [実績] → [チーム] → [お客様の声] → [サービス] → [料金] → [FAQ] → [アクセス] → [お問い合わせ] → [Footer]
```

良い例：
```
[Hero — 価値提案 + CTA]
[サービス概要 — 事業内容 + 強みを統合]
[実績数字 — 薄いバンドで3つの数字]
[お客様の声 — カード2〜3枚]
[About — 会社概要 + 理念 + チームを統合]
[CTA — お問い合わせ誘導]
[Footer — アクセス + 基本情報]
```

### 情報に強弱をつける

全セクションが「全幅 + 見出し + 本文」の同じパターンの繰り返しにしない。
情報の重要度に応じてセクションの面積・密度・レイアウトパターンを変える。

| 情報の重要度 | 表現方法 |
|---|---|
| 最重要（Hero・メインCTA） | 大きく、余白多め、画面占有率高い |
| 重要（サービス・強み） | カード型・グリッドで密度ある表現 |
| 補強（実績数字・受賞歴） | 薄いバンド・アクセント的に |
| 信頼（お客様の声・パートナー） | 引用 + 写真、コンパクトに |
| 行動喚起（CTA） | シンプルに1行 + ボタン |
| 補足（FAQ・アクセス） | 畳める構造、または Footer に統合 |

### ページ全体のリズムを設計する

セクションの「密度」「レイアウトパターン」「高さ」を意図的に変え、緩急をつける。
同じパターンが3回以上連続しないようにする。

リズムの例：
```
[大きく・余白多め] Hero — 呼吸
[密度あり・カード] サービス — 情報
[薄い・数字だけ]  実績 — アクセント
[引用・写真あり]  お客様の声 — 信頼
[シンプル・1行]   CTA — 行動
```

### ストーリーで考える

ページを「情報のリスト」ではなく「説得のシナリオ」として設計する。
ユーザーがスクロールしながら辿る心理の流れ：

```
興味（Hero: この会社/サービスは何？）
  ↓
理解（サービス: 何を提供してくれるのか）
  ↓
納得（実績・強み: なぜここを選ぶべきか）
  ↓
信頼（お客様の声: 本当に大丈夫か）
  ↓
行動（CTA: 問い合わせよう）
```

この心理フローに沿わない情報配置は、ユーザーの離脱につながる。

## 設計根拠の記録

セクション構成・順序の根拠をHTMLコメントに記録する。
後続フェーズのデザイナー・エンジニアが「なぜこの構成か」を理解するための引き継ぎ資料となる。

```html
<!--
  WIREFRAME STRATEGY: [サイト名]

  ターゲット: [誰が、どんな状況で見るか]
  主目的: [ユーザーに何をしてもらいたいか]

  ページストーリー:
  - [心理フロー: 興味→理解→信頼→行動 等]

  セクション設計:
  1. Hero — [なぜこのヒーロー構成か]
  2. [セクション名] — [目的 + なぜこの位置か + 統合した情報]
  3. [セクション名] — [目的 + なぜこの位置か]
  ...

  リズム設計:
  - [大→密→薄→引用→シンプル 等の緩急パターン]

  情報の統合:
  - [何と何を1セクションにまとめたか、その理由]
-->
```

## 絶対ルール（違反禁止）

| # | ルール |
|---|---|
| 1 | **グレースケールのみ使用**。色の判断をこのフェーズに持ち込まない |
| 2 | **CSSは全てインライン記述**（`style=""`属性のみ）。`<style>`タグ・外部CSSファイル禁止 |
| 3 | **単一HTMLファイルで完結**。複数ファイル分割禁止 |
| 4 | **外部リソース読み込み禁止**。CDN・Google Fonts・外部画像URL禁止 |
| 5 | **JavaScriptは禁止**。`<script>`タグ禁止 |
| 6 | **`<link>`タグ禁止**（faviconも含む） |
| 7 | ファイル名は `wireframe.html` とする |
| 8 | **ファイルは必ず上書き（write）する。追記（append）禁止。** 修正・再生成時も、完全なHTML文書1つだけで `wireframe.html` を上書きすること。`<!DOCTYPE html>` が2回以上出現するファイルは壊れたファイルである |

## カラーパレット（グレースケールのみ）

| 用途 | カラー |
|------|--------|
| 背景（メイン） | `#FFFFFF` |
| 背景（セクション交互） | `#F5F5F5` |
| テキスト（メイン） | `#333333` |
| テキスト（サブ） | `#666666` |
| テキスト（補足） | `#999999` |
| ボーダー・区切り線 | `#DDDDDD` |
| プレースホルダー背景 | `#E0E0E0` |
| CTA・重要要素 | `#333333`（白文字） |

> 色を排除することで、レビュー時に「構造と情報の優先度」だけに集中できる。
> ただし `DESIGN.md` に記載されたレイアウトの好み（余白感・密度・画像比率）は反映する。

## HTML構造ルール

### 基本テンプレート

```html
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>[サイト名] - ワイヤーフレーム</title>
</head>
<body style="margin: 0; padding: 0; font-family: -apple-system, BlinkMacSystemFont, 'Hiragino Sans', sans-serif; background-color: #FFFFFF; color: #333333;">
  <!-- WIREFRAME STRATEGY コメント -->
  <!-- セクションを順番に配置 -->
</body>
</html>
```

### セクション構造

- 各セクションは `<section>` タグで囲み、`id` 属性を付与する
- セクションごとに `style=""` でレイアウトを完結させる
- **各セクションの冒頭にHTMLコメントで「このセクションの意図」と「統合した情報」を記録する**
- ネストは最大4階層まで

```html
<!-- 意図: サービス内容と強みを統合。ユーザーが「何を得られるか」を一目で理解できるようにする -->
<section id="services" style="...">
  ...
</section>
```

## コンテンツルール

### テキスト

- 見出し・CTA・キャッチコピーは**実際に近い文言**を入れる（Lorem ipsum 禁止）
- 本文テキストは内容の方向性が分かる程度に記載する（完璧な原稿は不要）
- **文字量を実際に近づける**ことで、レイアウトの現実性を検証する

### 画像プレースホルダー

```html
<div style="width: 100%; height: 400px; background-color: #E0E0E0; display: flex; align-items: center; justify-content: center; border: 1px dashed #CCCCCC;">
  <span style="color: #999; font-size: 14px;">[画像: ヒーロービジュアル — 事業の雰囲気が伝わる写真]（1440×800）</span>
</div>
```

プレースホルダーには**どんな画像を入れるべきか**を具体的に記述する。
「画像」とだけ書くのではなく、「笑顔のスタッフが作業している写真」のように意図を伝える。

### 画像プレースホルダーの配置義務（空欄禁止）

- `id="hero"` のセクションには、必ず画像プレースホルダーを 1 つ入れる。
- `context.md` / `requirements.md` から「写真・図・実物イメージ」が必要になりそうなセクションでは、空欄にしない（必ず placeholder を置く）。
- カード型セクション（複数項目を並べる構造）では、各カードに「画像（またはサムネ）枠」を用意し、placeholder を入れる。

サムネ用プレースホルダー例：

```html
<div style="width: 100%; height: 160px; background-color: #E0E0E0; display: flex; align-items: center; justify-content: center; border: 1px dashed #CCCCCC;">
  <span style="color: #999; font-size: 14px;">[画像: 提供価値を象徴する図/写真]</span>
</div>
```

### アイコン

シンプルなSVGで直接記述する。外部アイコンフォント禁止。

## フォントサイズの基準

| 用途 | サイズ |
|------|--------|
| ヒーローキャッチ | 48px〜72px |
| H1（ページタイトル） | 36px〜48px |
| H2（セクションタイトル） | 28px〜36px |
| H3（サブタイトル） | 20px〜24px |
| 本文 | 16px（最小14px） |
| キャプション・注釈 | 12px〜13px |

## レイアウトルール

- **最大コンテンツ幅:** `max-width: 1200px; margin: 0 auto;`
- **セクションパディング:** `padding: 80px 40px;`（PCビュー基準）
- **グリッドはFlexboxで表現**
- **スペーシング（8の倍数系）:** `8px / 16px / 24px / 32px / 48px / 64px / 80px / 120px`

### レスポンシブ

ワイヤーフレームはPCビューのみで出力する。
ただし、レスポンシブで崩れにくい構造を意識する:

- `max-width` + `margin: auto` でセンタリング
- `flex-wrap: wrap` でカード等が折り返せる構造にする

## 出力チェックリスト

出力前に以下を必ず確認する。

**構造・ストーリー:**
- [ ] WIREFRAME STRATEGY コメントが冒頭にあるか
- [ ] 各セクションに意図コメントがあるか
- [ ] 「要件の項目数 = セクション数」になっていないか（情報が統合されているか）
- [ ] 同じレイアウトパターンが3回以上連続していないか（リズムがあるか）
- [ ] セクションごとに面積・密度が変わっているか（強弱があるか）
- [ ] ページ全体がストーリー（興味→理解→信頼→行動）として成立しているか
- [ ] CTAの配置が要件定義の目的に沿っているか

**コンテンツ:**
- [ ] グレースケール以外の色を使っていないか
- [ ] テキストが Lorem ipsum ではなく実際の内容に近いか
- [ ] 画像プレースホルダーに具体的な画像意図が書かれているか
- [ ] 画像領域があるセクションでは placeholder が入っているか（空欄禁止）

**技術:**
- [ ] `<style>`タグが存在しないか
- [ ] `<link>`タグが存在しないか
- [ ] `<script>`タグが存在しないか
- [ ] 外部URLが含まれていないか
- [ ] 全てのCSSが `style=""` 属性で記述されているか
- [ ] 単一ファイルで完結しているか
- [ ] ファイル名が `wireframe.html` になっているか
- [ ] `<!DOCTYPE html>` がファイル内に1回だけ存在するか（複数回 = 追記ミス）
